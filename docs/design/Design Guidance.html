Tenets

Every design decision in this product should pass through these four filters:

1. Family tool, not medical record

We write "Rick" and "Linda," not "Patient" and "Caregiver." We show recovery progress, not medical-record numbers. MRN, height, weight, code status belong in the ER handoff PDF — not the home screen. If a field feels like it came from an EHR intake form, it probably shouldn't be in the UI.

2. Calm when calm, assertive when not

Default surfaces are warm and low-contrast. Red, bold, and filled are earned states reserved for things that actually matter right now: a life-threatening allergy, a missed anticoagulant dose, an active MPA. Overusing urgency destroys our ability to signal it when it counts.

3. Every stakeholder sees a different app

The nav, the write affordances, and the primary surface change by role. Linda administers, Amy stewards the record, Rick does his own check-ins, Ben is the technical operator, aides are scoped. Don't hide things from people — give them a UI that fits their job.

4. Writes are first-class citizens

Every action (Mark taken, Log vitals, Check-in, Log fall, Add diagnosis, Verify allergy) flows through the shared useCareLog store, is attributed to a stakeholder, and surfaces somewhere meaningful. "What did I tap?" should always be answerable by scrolling the activity feed.

The visual system

Everything below already exists in the codebase. The job for subsequent modules is to reuse these, not to invent new ones.

Category tints (the spine of the visual language)

Each domain has a pastel background + darker ink + matching edge colour. Define via CSS variables (--cat-{name}-bg, --cat-{name}-ink, --cat-{name}-edge). Use the .cat-surface.cat-{name} class on any card that represents that domain.

allergy
Allergies, fall history, life-threatening, incidents
surgery
Surgical history, hardware, implants, documents
mpa
Medical POA, advance directive, legal authority
meds
Medications, pharmacy, supply, dose logs
people
Family, providers, aides, contacts
vitals
Readings, sparklines, orthostatic checks
recovery
PT, check-ins, recovery progress, BLT

Semantic states (earned colours)

These are not category tints — they are meaning. Don't use them for decoration.

  • critical — red. Life-threatening allergy, missed anticoagulant, active incident, overdue >2h.
  • warning — amber. Trending wrong, due soon, low supply, unverified important fact.
  • ok — green. Taken, verified, stable, no concerns.
  • info — blue. Neutral status, scheduled, on-track.

Type

  • t-display / t-h1 / t-h2 / t-h3 — never add new sizes; pick the closest.
  • t-body / t-body-sm — narrative text.
  • t-label — the small-caps kicker above every header.
  • t-meta / muted-2 — supporting/secondary info.
  • .mono — only for values that look wrong in a proportional font (BP, times, phone numbers, MRN).

Components we already have (use these first)

  • ScreenHeader — every inner screen; kicker + title + back + optional trailing action.
  • HeroCard — identity + recovery ring. Surfaces Rick's "who he is right now."
  • CategoryCard — pastel tile that links to a domain. Has primary/secondary lines, badge, chips.
  • VitalsTile — value + unit + sparkline + trend note. Anywhere we show a number over time.
  • MedCard — pill glyph + dose + supply bar + state chip. Never rebuild for a one-off.
  • AllergyRow — warmer list row, auto-elevates on life-threatening.
  • Sheet — the bottom-sheet modal scaffold. All logging forms inherit from this.
  • TimelineItem — the "Now" anchor and vertical today-plan row.
  • RecoveryRing — progress ring around any avatar or count.
  • SectionLabel — the spaced uppercase row that divides every screen into blocks.
Anti-patterns to avoid
  • Dense 3-column clinical strips (MRN / height / weight / code). We removed these on purpose.
  • Gradient backgrounds as decoration. We only use them for earned urgency (ER banner) and the hero card.
  • Stock emoji in UI. If you need a symbol, use the Lucide-style icon set or a coloured dot.
  • Bright brand accents on every card. We commit to pastel surfaces; accent colour is rare and intentional.
  • "Open details" buttons. The whole card is clickable; the chevron signals it.

M02Onboarding & household setup

The hardest module. How does a family get from zero to a populated record? This is where we either earn a year of use or lose them in twenty minutes. Don't treat it as forms.

Frame the story, not the fields

Frame each step around why it matters — "So the ER knows what he can't take" (allergies), not "Add allergies." The first screen after install shouldn't ask Linda to type; it should ask "who is the person you're caring for?" and give her scannable options.

Three on-ramps, not one

  • Fast path (90 sec): name, age, photo, top allergy, top 3 meds, one emergency contact. Good enough for the fridge magnet.
  • After a hospital visit: upload discharge summary PDF → OCR or AI extract → review/accept.
  • Invited by another family member: Amy or Ben already set it up; Linda just accepts an invite and gets a populated record.

Invitations & role selection

A single "Invite someone" sheet with role picker (Primary caregiver · MPA holder · Family · Paid aide · Patient). Show what each role will see and be able to do, in plain language, from Rick's perspective. "Amy will be able to: read everything, add and verify records, receive ER alerts. Amy will not be able to: log Rick's daily vitals or medications."

Reuse

Components to reuse
  • CategoryCard tiles as role pickers — each role tinted by the domain they own.
  • Sheet bottom-sheets for "Add allergy," "Add contact," "Invite family."
  • HeroCard in its "pre-recovery" variant: no ring, just name + photo + "recovering from X."
  • The ER handoff PDF preview as the final onboarding screen — "here's the version of Rick you just built."

M03Alerts, notifications & "something's wrong"

Module 01 is a calm app. Module 03 designs the uncalm state. Get this wrong and we become a notification app that nobody trusts.

Alert taxonomy — three tiers, not a spectrum

  • Tier 1 — critical: missed anticoagulant dose >2 hours, life-threatening symptom flagged in check-in, fall with injury, vitals past a doctor-set threshold. Push notification with sound, red banner at top of app, ER button pulses.
  • Tier 2 — attention: overdue dose 30–120 min, vitals trending wrong over 3 days, unverified allergy, new document awaiting review, MPA expiration in 30 days. In-app amber chip, quiet push, shows up as a "Needs attention" count on Today.
  • Tier 3 — informational: "Amy added a note," "Dose logged by Aide." In-app only, shows in activity feed, never interrupts.

The assertive state, visually

When we need to get Linda's attention, the app doesn't just add a badge. Tier 1 does three things at once: (1) a critical-wash banner takes over the top of Today with the specific concern + a one-tap resolution, (2) the BottomNav's ER button pulses, (3) push notification with Rick's photo and one line — never the medical term; say "Rick's Apixaban is 2 hours overdue," not "Anticoagulant non-compliance."

Dismiss, snooze, resolve — different verbs

Alerts never just go away. Linda resolves them (with a log entry), snoozes them (with a reason and duration), or escalates them to Amy. "X" never appears on a critical alert — that's a trap.

Hard rules
  • Tier 1 never appears for something we can't name precisely. Never "Rick needs attention" — say what.
  • No more than one Tier 1 alert visible at once. Queue them; show next when current is resolved.
  • Tier 3 never becomes Tier 2 from accumulation. "5 unread updates" is not an emergency.
  • Aide and Rick see fewer Tier 2 alerts than Linda/Amy. Scope by role — don't spam the scoped user.

Reuse

Components to reuse
  • The critical-wash ER button on Today is already the pattern — scale it up for Tier 1 banners.
  • Chips (chip.critical / .warning) are the in-line alert language.
  • Activity feed entries double as Tier 3 delivery — no new surface needed.
  • Bottom-sheet Sheet for the resolution/snooze/escalate picker.

M04Amy's verify & edit loop

Amy is the record's steward. She adds, edits, and verifies. This module is what separates our app from a generic patient portal.

"Needs attention" as her Today surface

Amy's Today should lead with a queue: "2 unverified allergies," "New document uploaded by Linda — review?", "Dr. Kim's phone number hasn't been checked in 8 months." The queue is prioritized, dismissable, and generates a badge on the Records tab when populated.

Verify is a three-state toggle

Every factual record has one of:

  • Unverified — added by a family member, not yet confirmed by a clinician. Shows amber chip.
  • Verified — Amy (or a provider) confirmed directly, with attribution and date.
  • Disputed — conflicting entry (e.g. Rick says allergic to shellfish, ER records don't list it). Needs resolution.

Disputed is the state we haven't designed. It's a neutral-grey chip that opens a sheet listing the two sources side-by-side with "Accept A," "Accept B," "Keep both."

Inline edit, with attribution

Amy taps a value (phone number, reaction type, provider org). The value becomes an input in place. Saving logs "Amy updated X from Y to Z" to the activity feed. No modal, no form screen — the record is the form.

The upload pipeline (docs → structured data)

Amy uploads a PDF. The app shows: document preview, extracted candidate fields (diagnoses, meds, dates), and a review row per candidate — "Add to Rick's record?" Reject/Edit/Accept per row. Never auto-accept; never force her to fill the form manually.

Components to reuse
  • AllergyDetailSheet already has the verify/edit pattern — generalize to "RecordDetailSheet."
  • ActivityFeed is where every edit surfaces. No new "audit log" UI needed.
  • CategoryCard with a badge ("2 unverified") drives the Records landing.

M05Rick's patient view

Currently Rick sees a trimmed version of Linda's app. That's the MVP. The real patient view is a different product.

What Rick's Today should actually be

Strip it down to: a warm greeting, one primary action (the next thing he should do — take a pill, do a check-in, rest), and his recovery progress. No vital tiles. No timeline. No activity feed. The more he sees, the less likely he uses it.

Readable from across the room

Text baseline should be 16px at minimum, and the primary action button should never be smaller than 56px tall for Rick. Contrast ratios should pass WCAG AAA, not AA. Sparklines are excluded from his view — they need scrutiny he won't give.

One-handed, home-row thumb zone

Every tappable element Rick cares about sits in the bottom half of the screen. The bottom nav we have is right; the quick-action row we added for caregivers stays in the thumb zone.

Plain language, everywhere

  • "Blood thinner" not "Apixaban"
  • "For pain" not "Oxycodone"
  • "Surgery healing" not "Post-op status"
  • "How you felt yesterday" not "Check-in history"
Hard rules
  • Rick never sees MPA status as a warning — he's the holder when it's inactive. Frame as "You decide."
  • Rick never sees his own fall history on Today. It's for the caregivers.
  • Rick's write affordances only: mark-taken, 30-sec check-in. Everything else is view-only.

M06Handoff family

The ER PDF is one of several handoffs. Each has a different audience, density, and tone.

ER handoff
Dense, clinical, single-page. For a 15-second read by a stranger. Already built.
Surgeon follow-up
Meds + symptoms + PT adherence + questions Linda wrote down. 1–2 pages. Generated for a scheduled visit.
Aide shift brief
Today's schedule + what to watch for + "don't do these things." One page, friendly, photo-led.
Rehab admission
Full history export — allergies, hardware, current meds, recent vitals, providers. 3–4 pages, clinical.

Design rules across all handoffs

  • Review before generate. The user always sees what will be included and can toggle sensitive sections (insurance, MPA, mental health) before the PDF exists.
  • One font, one weight, one accent. Handoffs aren't the place for our pastel palette. Black-on-white with one critical red for the "cannot take this" rows.
  • Attribution footer. "Prepared by Linda · generated Apr 19, 2026 · sources: Brigham discharge 3/18, family-reported." Lets the reader trust or discount.
  • QR code in the footer. Points to a time-limited web view of the same data so a provider can update us without typing.

Reuse

Components to reuse
  • ERPDFPreview is the scaffold — parametrize the sections by handoff type.
  • Section chooser UI from ER handoff (the "Included by default" + "Optional" pattern) generalizes.
  • CategoryCard on a "Handoffs" landing screen — one tile per handoff type, each linked to its generator.

M07Activity feed, permissions & the family graph

We have useCareLog and a Today feed. The full module is the family-graph layer on top: who sees what, who can do what, and how the history of changes is visible to everyone.

The activity feed as the app's nervous system

Every write — every log, every edit, every verification, every upload — flows through useCareLog with a kind, timestamp, actor, and detail. The full feed is a filterable timeline (by kind, by person, by date range) and is where Amy audits Linda's care and vice versa.

Permissions, not roles

Role presets (Primary / MPA / Family / Aide / Patient) are starting points. The real model is per-capability: can this person read / write / verify / share each of the eight domains? Build the matrix once; present it as simple presets 95% of the time and as the full grid only when Linda wants to customize an aide's scope.

Invitation & exit flows

  • Adding a person: email/phone → role preset → welcome screen tailored to what they'll see.
  • Removing a person: their history is preserved, their future access is revoked, they get a polite notice. Never silently.
  • Transferring primary: if Linda can no longer be primary (hospitalized herself), Amy can accept primary with a signed acknowledgement.

The MPA activation audit trail

Already in Module 01. The full module extends it to: who was present when it was activated, what specific decisions were made under activation (DNR consent, surgical consent, discharge disposition), and the deactivation record. This is the most legally sensitive part of the app — it deserves its own clean, formal surface distinct from the warm everyday UI.

Components to reuse
  • ActivityFeed row component scales to a full-screen filterable timeline.
  • MPAScreen's activation history is the template for the decision audit trail.
  • Per-role nav config (NAV_CONFIGS) is the canonical pattern for role-based UI divergence.

Closing note

The goal of every module that follows is not to add new surfaces. It's to deepen the few we have. The Today screen, the Records tab, the activity feed, the handoff PDF — these are the four things a real family will interact with 90% of the time. If a module can't be expressed as changes to those four surfaces, ask hard whether it belongs.
